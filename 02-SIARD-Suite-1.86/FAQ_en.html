Where can I get the current version of SIARD Suite?

SIARD Suite was developed by the Swiss Federal Archives and is furnished free of charge to any interested party.

The current version can be ordered from the SIARD Download page.

Why can I not open a SIARD file using WinZip?

SIARD files are stored in the ZIP64 format as described in the documentation of the SIARD format. This format is a successor of the common ZIP format (version 4.5) developed by the company PkWare. Most ZIP utilities available today only support the common ZIP (32-Bit) format. The ZIP32 format is not suitable for the archival of databases, because it is limited to the maximum size of 4GB and 64k contained files.

SIARD files can be opened using PKZIP from PkWare.

There is an open source command-line utility published by Enter AG under http://sourceforge.net/projects/zip64file/ which permits extracting and compressing of ZIP64 files.

Why do I need to always archive the whole database? Why can I not select the tables I want to archive?

The "whole database" is the collection of all objects that the database user has read access to. A general way of designating tables to be archived or skipped would be very cumbersome. The access control subsystem of the database system does a much better job of presenting a consistent view to each user.

To exclude some tables from archival just create an archival user with read access limited to the tables that should be archived. Use this archival user for downloading the database using SiardFromDb or SiardEdit.

Why can I not select the table columns to be archived?

The most flexible selection is very complex to describe. Instead we recommend to describe the material to be archived in VIEWS using SQL and "realise" those view in separate tables to which the archival user has read access.

Why does SIARD not archive CHECK constraints, TRIGGERS, VIEW queries oder ROUTINE sources?

The SIARD format was designed for long-term database preservation. The database objects CHECK, TRIGGER, VIEW and ROUTINE were included because they are part of the SQL standard and other programs producing SIARD files may want to fill them for informative purposes. SIARD Suite created those database objects to the extent they are accessible in the database system through JDBC.

To serve its main purpose of long-term preservation of database data SIARD Suite makes sure that the content of the base tables and the metadata describing the database structure is preserved most faithfully. Preserving action metadata (TRIGGER, ROUTINE) that change the data and queries that are formulated in the idiosyncratic SQL dialect of the database system (CHECK constraints, VIEW queries) is considered of secondary importance and are only archived if they are accessible as long as no existing database system supports the standards that should be able to run on all other SQL:1999 conformant database systems.

We want to archive data rather than code.

Otherwise we could start archiving all of the applicatory code (in C/C++, JAVA,Python, VB.NET, Ruby ...)!

Why does SIARD slow down when downloading large databases?

SIARD Suite was designed to be able to archive any database of any size with limited JAVA memory. (There is one exception: The whole structural metadata must fit into memory. If you have a database with very many e.g. thousands of tables and columns, you may have to start the JVM with a larger heap.)

SiardEdit does offer the download functionality but is not suited for the download of large databases because the output is displayed in GUI window, which fills up the JAVA heap and makes the process slower, the larger the displayed output has become.

This is the reason why the command-line utilities SiardFromDb and SiardToDb were developed. These utilities are not subjected to this problem of a growing heap, because they write their output to stdout or stderr.

With these utilities a constant download speed is achieved which does not depend on the size of the database but is mainly limited by the bandwidth of the network and the speed of the database server.

Why does SIARD not store any primary keys, foreign keys or VIEWS when archiving MS Access files?

SIARD Suite can only archive what the database system permits to extract. MS Access has no concept of a primary key but treats all candidate (unique) keys the same. In many cases SIARD Suite succeeds in guessing the name of the primary key (e.g. the standard name "PrimaryKey", but that differs from on Windows locale language to the other!) but success cannot be garanteed.

If the system tables MSysObjects, MSysQueries and MSysReferences could not be made accessible to SIARD Suite - e.g. for .accdb files - then information about foreign keys and VIEWS cannot be archived.

What kind of support is available for SIARD Suite?

The Swiss Federal Archives have authorized the developers of SIARD Suite at Enter AG in Zurich to offer support contracts to other institutions.

Can SIARD Suite be translated to other languages?

Technically SIARD Suite can be augmented to support any number of additional user languages. An application for support of a new language must be submitted to the Swiss Federal Archives.

Why does SIARD Suite not support any other database systems?

In principle there is no limit to the number of database systems that could be supported by SIARD Suite as long as a JDBC driver exists, to access the database. Unfortunately the various database systems support very different, incompatible versions of SQL-92. (In 2010 there does not exist a single database system yet which conforms to the SQL:1999 standard! Obviously the producers of these systems are motivated to chain their users to their products.)

Therefore every new database system must be integrated to SIARD Suite "manually". In spite of separation of generic database code and the proprietary databases no clean global interface to "external" new database systems can be defined, that will handle all specialties of real database systems.

The Swiss Federal Archives keep adding support for new database systems. If you have a particular need, please let them know!
